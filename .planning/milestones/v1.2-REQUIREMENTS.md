# Requirements Archive: Glanceus v1.2 SQL Data Access and Visualization Expansion

**Defined:** 2026-03-23
**Core Value:** Users can complete auth -> fetch -> parse -> render through config-only integrations without backend hardcoding.

## v1 Requirements

### SQL Step Contracts
- [x] **SQL-01**: User can declare a `use: sql` step in Integration YAML with a connector profile and credentials resolved from `secrets` channels. (Phase 8)
- [x] **SQL-02**: User can execute fully user-authored SQL query text (script-like control), and have non-SELECT/high-risk operations detected by static AST analysis and routed to authorization before execution. (Phase 8)
- [x] **SQL-03**: User can use multiple SQL connector patterns in one contract model (at least local SQLite and server-side PostgreSQL profiles). (Phase 9)
- [x] **SQL-04**: User can rely on default SQL guardrails (timeout, max row limit, deterministic truncation behavior) with per-source overrides. (Phase 8)
- [x] **SQL-05**: User can receive deterministic `error_code` responses for SQL connect/auth/query/guardrail failures. (Phase 8)
- [x] **SQL-06**: User can run SQL steps without database credentials leaking into logs, API payloads, or persisted runtime artifacts. (Phase 8)

### Normalized SQL Integration Data
- [x] **DATA-01**: User can receive SQL query output as normalized Integration Data (`rows` + typed field metadata) that is stable across supported connectors. (Phase 9)
- [x] **DATA-02**: User can rely on deterministic serialization for SQL-native values (`decimal`, `datetime`, `null`, `bytes`) before template/chart consumption. (Phase 9)
- [x] **DATA-03**: User can consume SQL execution metadata (`row_count`, `duration_ms`, truncation flag) for diagnostics and card rendering. (Phase 9)
- [x] **DATA-04**: User can reference SQL outputs through existing template expression channels without moving business logic into frontend widgets. (Phase 9)

### Chart Widgets
- [x] **CHART-01**: User can render SQL-backed line charts in Bento Cards. (Phase 10)
- [x] **CHART-02**: User can render SQL-backed bar charts in Bento Cards. (Phase 10)
- [x] **CHART-03**: User can render SQL-backed area charts in Bento Cards. (Phase 10)
- [x] **CHART-04**: User can render SQL-backed pie charts in Bento Cards. (Phase 10)
- [x] **CHART-05**: User can render SQL-backed tabular views as a chart-compatible widget for dense result inspection. (Phase 10)
- [x] **CHART-06**: User can rely on deterministic empty/loading/error fallback states for all new chart widgets. (Phase 10)
- [x] **CHART-07**: User can configure x/y/series/value field mapping from normalized Integration Data with runtime schema validation. (Phase 10)

### Usability and Diagnostics
- [ ] **UX-01**: (Dropped/Deferred) User can preview SQL sample rows and inferred field schema. (Phase 11 removed)
- [ ] **UX-02**: (Dropped/Deferred) User can diagnose SQL/chart setup failures through stable i18n messages. (Phase 11 removed)
- [ ] **UX-03**: (Dropped/Deferred) User can apply dashboard-level filter values to SQL query inputs. (Phase 11 removed)
- [ ] **UX-04**: (Dropped/Deferred) User can use SQL+chart workflows without duplicate SWR write paths. (Phase 11 removed)

### Security and Storage Trust
- [x] **SEC-HTTP-01**: User can rely on loopback/private target classification for HTTP steps. (Phase 7)
- [x] **SEC-HTTP-02**: User can authorize HTTP private-target access through policy-driven trust prompts. (Phase 7)
- [x] **SEC-DB-01**: User can rely on deterministic identity for database trust rules (capability/source/target). (Phase 7)
- [x] **STORAGE-TRUST-01**: User can persist trust decisions in SQLite with source-scoped cascade deletion. (Phase 7)
