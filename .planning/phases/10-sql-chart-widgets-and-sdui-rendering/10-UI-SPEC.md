## Spacing Scale

Declared values (allowed set only):

| Token | Value | Usage |
|-------|-------|-------|
| xs | 4px | Micro gaps, icon-text separation, compact inline spacing |
| sm | 8px | Small control padding, chip/badge spacing, stacked micro-layouts |
| md | 16px | Default widget internal gap, card body spacing, compact chart legend/list spacing |
| lg | 24px | Default section spacing inside a card, chart header-to-body spacing, grouped empty/error state spacing |
| xl | 32px | Major separation inside larger widgets and table/chart composites |
| 2xl | 48px | Card-level breathing room for high-emphasis states and page-level interior groups |
| 3xl | 64px | Reserved for dashboard-level separation or oversized focal layouts only |

Phase 10 scaling contract:
- The canonical component size vocabulary is `sm | md | lg | xl`.
- The canonical density vocabulary is `compact | normal | relaxed` via the existing `--qb-density` system.
- New chart widgets must scale spacing through these two existing mechanisms instead of inventing widget-specific one-off spacing classes.
- Reuse the repo‚Äôs existing density-backed spacing tokens from `src/index.css`:
  - `qb-gap-1` = 4px base x density
  - `qb-gap-2` = 8px base x density
  - `qb-gap-4` = 16px base x density
  - `qb-gap-6` = 24px base x density
  - `qb-gap-8` = 32px base x density
  - `qb-gap-12` = 48px base x density when a larger visual break is required
- Reuse the existing semantic mapping pattern from `commonProps.ts` while constraining implementation to the approved spacing set above:
  - layout spacing: `sm -> 8px`, `md -> 16px`, `lg -> 24px`, `xl -> 32px`
  - micro spacing: `sm -> 4px`, `md -> 8px`, `lg -> 16px`
- For Phase 10 chart widgets, expose spacing only through semantic size or shared layout spacing props. Executor must not hardcode arbitrary values such as `gap-[10px]`, `gap-[12px]`, `p-[13px]`, or `p-[20px]`.
- Chart card internals must preserve the existing card shell rhythm:
  - header height stays fixed at the shared `--qb-card-header-height` token
  - card padding continues to use `qb-pad` / `qb-pad-x` / `qb-pad-y`
  - chart content regions use approved `qb-gap-*` tokens that resolve to the allowed spacing set so density changes scale the whole widget consistently

Size-to-spacing behavior for new chart widgets:
- `size="sm"`: use micro rhythm (4px / 8px) for legends, labels, and compact fallback copy.
- `size="md"`: use default rhythm (8px / 16px) for standard dashboard cards.
- `size="lg"`: use relaxed rhythm (16px / 24px) for larger chart cards and table variants.
- `size="xl"`: reserve for high-emphasis metric or hero-like chart presentations; spacing may step up to 32px, and only to 48px when separating the primary chart from secondary metadata blocks.

Exceptions: touch and click targets may round up to 32px minimum for `sm`, 48px for `md`, and 48px minimum for destructive or primary actions when embedded in chart cards; do not shrink icon-only interactive targets below 32px.

## Typography

| Role | Size | Weight | Line Height |
|------|------|--------|-------------|
| Label | 12px | 600 | 1.2 |
| Body | 14px | 400 | 1.5 |
| Heading | 18px | 600 | 1.2 |
| Display | 30px | 600 | 1.2 |

Phase 10 typography must support generic size-based scaling using the repo‚Äôs existing size pattern rather than introducing a separate chart-only type ramp.

Canonical typography rules:
- Allowed font sizes across the entire document: `12px`, `14px`, `18px`, and `30px` only.
- Allowed weights: `400` and `600` only.
- Default body text is `14px/1.5`.
- Labels, captions, legends, and micro metadata use `12px/1.2` semibold or `12px/1.5` regular when dense data readability matters.
- Card titles and chart section headings use `18px/1.2` semibold.
- Large numeric metrics or prominent chart summary values use `30px/1.2` semibold.

Generic size-scaling contract for widgets:
- Reuse the existing semantic size tokens from `commonProps.ts` and current widgets:
  - `sm -> 12px` for compact labels and control text
  - `md -> 14px` for default widget body text
  - `lg -> 14px body text with 18px heading/title treatment`
  - `xl -> 30px` for headline metrics only
- Reuse current component precedents already in repo:
  - buttons: `sm`, `default`, `lg`, `icon` in `ui/button.tsx`
  - badges/actions/progress: `sm | md | lg | xl`
  - metric number: `text-3xl md:text-4xl lg:text-5xl` in `ui/micro.tsx`
- For Phase 10 charts, the executor should map widget `size` to typography as follows:
  - `sm`: labels/legend/table cell text = 12px; title remains 14px if present
  - `md`: labels/body/table cell text = 14px; title = 18px
  - `lg`: labels/body/table cell text remains 14px for readability consistency; title and emphasized sectional text = 18px
  - `xl`: reserve for display metrics, not general chart body copy
- Axis labels, legends, table cells, fallback descriptions, and inline help must all scale from the same semantic size prop so typography and spacing move together.
- Do not introduce arbitrary font sizes like `13px`, `15px`, `16px`, `17px`, `20px`, or chart-library inline typography styles unless there is a renderer constraint that cannot be expressed through existing classes.

Primary visual-priority rule for the main chart card:
- The chart plot area is the focal point. Within each primary chart card, visual attention must resolve in this order: card title/context first, plotted data second as the dominant occupied area, and supporting legend/table/error metadata third. Use the 18px heading only once at the card header, keep supporting copy at 12px or 14px, and reserve 30px display treatment for a single summary metric when that metric is the chart‚Äôs explicit hero value.

- If custom `colors` are supplied in chart config, they must remain light/dark safe and should be mapped through this approved palette unless a later phase explicitly expands the token system.

## Copywriting Contract

| Element | Copy |
|---------|------|
| Primary CTA | Refresh chart |
| Empty state heading | No chart data available |
| Empty state body | This widget has no rows to visualize yet. Refresh the source or adjust the query to return chartable results. |
| Error state | This chart cannot be shown right now. Check the SQL source status or fix the widget field mapping, then refresh the card. |
| Destructive confirmation | Remove chart widget: Remove this chart from the dashboard? This only deletes the card configuration and does not delete the SQL source. |

Additional deterministic copy rules for Phase 10:
- Invalid mapping/configuration state heading: `Invalid chart configuration`
- Invalid mapping/configuration state body: `One or more required fields are missing or incompatible for this chart type. Update the field mapping and try again.`
- Loading state headings remain chart-specific, but body copy must stay short and neutral; do not imply data loss.
- `Chart.Table` empty state uses the same empty-state heading/body as other chart widgets to preserve card-level consistency.
- Runtime errors must describe the problem and next step; never expose raw backend trace details or connector-specific internals.

Source notes:
- Empty/error/fallback semantics are locked by `10-CONTEXT.md` and `10-RESEARCH.md`.
- Specific copy above is defaulted in this contract because upstream artifacts defined behavior but not final wording.
